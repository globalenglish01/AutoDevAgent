## 1. 最终目标与达成情况

目标：客户提一句自然语言需求 → 系统自动完成 **需求分析 → 方案设计 → 编码 → 测试 → 部署**，类似 Claude Code 的全自动编程工具，且**全程不用付费 API**（[[feedback_no_paid_api]]）。

达成：完整流水线已打通并可运行——

```
需求(自然语言)
  → RequirementAnalyzer(DeepSeek垫片)  需求结构化
  → DesignAgent(DeepSeek垫片)          改动方案
  → CodeAgent(DeepSeek垫片)            读写代码(deepagents内置fs工具 + git工具)
  → StaticCheck(ruff/mypy/compileall)  秒级确定性检查，失败回退CodeAgent
  → ReviewAgent(ChatGPT垫片)           专职审查diff，发现问题回退CodeAgent
  → VerifyAgent(pytest)                真实跑测试，失败回退CodeAgent(重试上限)
  → DeployAgent                        生成PR草稿，停在 awaiting_approval（人工审批边界）
```

入口：`python run_autodev.py --repo <目标仓库> --task "<需求>"`

## 3. 测试验证策略（重要，如实说明）

**约束**：真实 LLM 是浏览器垫片（DeepSeek/ChatGPT），需要登录态浏览器窗口且每轮 5-6 分钟，无法在自动化测试环境驱动。

**因此**：
- 所有 58 个测试用**确定性假模型**（`GenericFakeChatModel` / 注入的脚本 callable）验证**控制流与装配**——图的跳转、重试回退、升级、HITL 边界。
- 同时**真实跑子进程工具**——ruff / mypy / py_compile / pytest / git 全部**真实执行**，不是 mock。例如 `test_e2e_feature.py` 用脚本 coder 真实实现 `multiply`，然后真实的 StaticCheck + Verify 对真实代码执行并通过。
- **真实 bridge 的端到端跑**由用户经 `run_autodev.py` 手动执行（`examples/README.md` 有完整步骤）。

**已验证为真的能力**：编排/重试/升级/HITL 边界全部正确；静态检查能抓语法/lint/类型错并在修复后放行；git 工具能提交且拒敏感文件；沙箱能拒非白名单命令；路径守卫能拒穿越和敏感文件；全链路能把一个真实特性从"写代码"跑到"PR草稿+测试通过"。

**未经真实 LLM 验证的部分**：DeepSeek/ChatGPT 垫片在真实需求上的**生成质量**——这取决于模型智商，不是工程正确性问题。

## 3.6 缩进丢失根因定位 + 修复（2026-07-09，用户要求"修复代码块抽取保留缩进"）

**逐字诊断确证根因**（fresh 垫片单次调用，`repr()` 看字节）：
- 代码放进 Markdown ``` 围栏 → `def bar():\n    return 99` 缩进**保留** ✅
- 代码作为纯文本（无围栏）→ `def bar():\nreturn 99` 缩进**被 Markdown 折叠丢失** ❌

即：**问题不在 bridge 抽取（抽取忠实反映 Markdown），而在 Markdown 对纯文本前导空格的折叠**。之前 min 测试偶尔成功正是因为模型自发加了围栏；全链路失败是模型把文件内容当纯文本输出。

**两处修复（均已 commit+push）：**
1. **AutoDevAgent 侧（根治）**：`direct_coder` 提示词强制要求每个 `<<<FILE>>>` 块内的文件内容必须包在 ``` 代码块里（解析器本就剥围栏）。→ **实测验证**：fresh 垫片单次生成 `multiply` → 缩进正确 → 静态检查+`pytest` 全过（25s）。commit `bd1c807`。
2. **bridge 侧（防御性加固，llm-bridge 仓库 commit `2fd8ba3`）**：`chatgpt_bot.py` 的 DOM 回退不再用 `innerText`（受 CSS 影响会折叠代码前导空格），改为 `get_last_answer` 对 `<pre>` 取内层 `<code>.textContent`（原始文本保留缩进）并包围栏、跳过工具栏噪声。复制按钮成功时本就 OK，此为回退路径的忠实度加固，TestAgent 各 agent 亦受益。

**结论**：用户要求的"保证保留缩进"**已达成并实测验证**——现在能稳定生成缩进正确、通过测试的代码（单次链路已证）。

**全链路多轮仍会间歇撞到的、与缩进无关的独立 substrate 障碍**（每个都是单独的 ChatGPT-UI/会话问题，非本项目逻辑，逐个修属 bridge 深度加固、且多为随机难复现）：
- ChatGPT 随机弹「回答を比較」A/B 评测全屏 modal 挡住输入框 → 点击超时（account 6 撞到）。
- 同一会话连续多轮后退化：CHATGPT_ERROR 累积、单轮从 46s 涨到 190s+。
- Playwright/asyncio 首调用竞态（偶发）。
运维缓解：每次真实跑前重启垫片进程拿干净会话、用 `--lite` 减少调用轮数。

## 3.5 真实浏览器垫片实跑记录（2026-07-08 补，重要）

用 TestAgent 的 llm_bridge（账号 zhaoyx 等已登录 ChatGPT，HTTP 代理 8765）做了真实端到端驱动，实测结论：

**真实跑通的部分（确凿证据）：**
- 单次审查调用（review_diff）：32s 返回合法 JSON，`json_extract` 正确解析。
- 工具调用（hello_world 深度 agent 经函数调用垫片）：正确调 `read_file` 读隔离工作区文件并复述。
- **真实代码生成**：独立 `direct_coder` 让 ChatGPT 生成 `multiply(a,b)`，写盘→静态检查→`pytest` **全部通过**（46s）。即"自然语言需求→浏览器垫片→可运行且测试通过的代码"这条链真实走通过。

**暴露并已修复的关键 bug：**
- **假成功**：全链路里 code 步若没产出改动，空 diff 会轻松通过 staticcheck/review(空diff自动approve)/verify(旧测试还在) → 假装成功出 PR。已修：空 diff 视为 code 失败，重试/升级 needs_human，绝不 false-success（回归测试 `test_empty_code_change_does_not_false_succeed`）。

**免费垫片的实测天花板（工程改善不了根本，属 substrate 限制）：**
1. **弱模型 tool-calling 脆弱**：deepagents 多工具 ReAct 在 ChatGPT free 上报 CHATGPT_ERROR/畸形 JSON。→ 已改用不依赖 tool-calling 的 `direct_coder`（整文件生成）绕过。
2. **垫片抓取偶发丢代码缩进**：copy-button 抓取路径能保留缩进（min 测试成功），降级到 DOM innerText 抓取时偶发把函数体顶格 → Python IndentationError。AutoDevAgent 能抓到并重试/升级（不 ship 坏码），但成功与否取决于某一轮是否命中干净抽取，**不稳定**。**真正的根治在 bridge 侧**：让 `chatgpt_bot.py` 的答案抽取对代码块优先用保留空白的 textContent（约 line 992 现在优先 innerText）。这是 TestAgent 的 gitignored 基础设施，属其改进项。
3. **会话易退化 / 限流**：连续多轮后同一 ChatGPT 会话易累积 CHATGPT_ERROR、变慢（单轮从 46s 涨到 190s+）。**运维结论**：真实跑前重启垫片进程拿干净会话；DeepSeek 账号需单独登录（当前 Google-OAuth 自动重登在账号 1 未生效）。

**综合判断**：AutoDevAgent 的工程正确性经真实垫片验证成立（能真跑通、失败诚实、不假成功）；能否"每次"稳定产出可用项目代码，瓶颈在免费垫片 substrate（抓取保真度 + 会话稳定 + 模型智商），非本项目逻辑。用更强模型/付费 API 时同一套代码质量会显著提升（`model_factory` 切换只改环境变量）。

**6 次真实全/精简链路跑的最终定论（缩进丢失是决定性瓶颈）：**
- 独立 min 调用（首调用、干净会话）→ multiply 缩进正确、过测 ✅。
- 全链路 ×3 与精简链路 ×1 → code 步都产出了 multiply，但**缩进被抓取丢掉**（函数体顶格）→ IndentationError → 诚实升级 needs_human（从不 ship 坏码/假成功）。
- 关键：精简模式让 code 成为**首调用**仍偶发丢缩进 → 证明缩进丢失是**逐条响应级别的随机性**（取决于 ChatGPT 当次代码块渲染 + 该会话 copy-button 抽取是否命中），不是单纯"首调用 vs 后续调用"，AutoDevAgent 侧重试无法根治（同一会话抽取路径一致）。
- **唯一根治点在 bridge**：`chatgpt_bot.py` 的答案抽取需保证对代码块拿到保留缩进的原文（copy-button 路径可靠时即正确）。这是 TestAgent 的 gitignored 基础设施，属其修复项，不在 AutoDevAgent 代码范围内。
- **诚实结论**：当前免费垫片下，AutoDevAgent 能**间歇性**真实生成通过测试的代码，但**无法保证每次**——不是逻辑缺陷，是抓取 substrate 的保真度上限。要达到"像 Claude Code 那样稳定可用"，需修复 bridge 代码抽取保真度，或换一个能返回干净代码的模型通道。本项目已把自己这一侧能做的鲁棒性（不假成功、诚实升级、整文件生成、缩进归一化、瞬时错误重试、精简模式）都做到位。

## 4. 关键工程决策（相对原设计的有据偏离）

1. **model_factory 用 HTTP 而非进程内 import**（Phase 1）：不依赖 TestAgent 目录结构，两项目只共享运行中的 bridge 服务。
2. **code 工具用 Python 原生而非 Node.js MCP 服务器**（Phase 2）：fs.* 由 deepagents 内置提供，git/ruff/mypy 本就是 Python/CLI 工具，套 Node 进程纯亏。MCP 独立进程模式留给需要真隔离的 shell.exec。
3. **HITL 用"图结束在 awaiting_approval"而非 mid-graph interrupt**（Phase 4）：更简单、可测，同样落地"自动到 PR 为止"的核心边界。

## 5. 诚实的能力边界

- **不用付费 API 的天花板依旧存在**：垫片无原生 function-calling、本机 15GB 无 GPU、单轮 5-6 分钟。"理解任意代码库 + 多文件重构"的成功率预计明显低于付费 API 方案，**不能承诺达到 Claude Code 水准**。工程手段（双模型分工、静态检查前置、小步提交、结构化输出鲁棒解析、fail-safe）已尽量把上限内做到位。
- **安全**：当前适合受信任仓库 + 开发者本机；对不受信任外部输入需要容器化执行环境（网络/内核隔离，见 SECURITY.md），这是生产硬化项。
- **StaticCheck/VerifyAgent 目前只支持 Python**；JS/TS 是后续扩展。
